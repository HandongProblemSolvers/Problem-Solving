**내가 이해한 문제내용**

Travel 
각 케이스마다 공항의 수가 정해지고
공항이 node라고 하면 각 edge 마다 비용과 소요시간이 존재.
비용에는 제한이 있음. 소요시간에는 없음. 비용이 같을 시에는
소요시간이 짧아야함.

 <hr/>
 
**접근 방식**

문제를 봤을 때는 
맨 처음에는 dfs를 생각했는데, 제한 시간이 10초나 되는 것을 보고
오히려 dfs는 아닐거야라고 생각했습니다.
그러면 dp일까 하고 생각했는데, 자연수 T의 개수를 모르니 
분명 엄청난 수의 case들이 있을거야라는 생각이 들어서..
greedy가 아닐까하는 생각으로 greedy로 접근했습니다.

 <hr/>
 
**어려웠던 점**

소요시간은 상관이 없으므로 비용 / 소요시간으로 먼저 정렬하고
출발지와 도착지 사이의 거리가 먼 순서로 정렬했습니다.  
각 노드마다 잇는 알고리즘을 구현했습니다.

그런데 각 ticket이 optimal하지 않았습니다.
예제는 맞았고 로직도 맞아보였는 데, 너무 간단한 예였던 것 같습니다.

어줍잖게 괜히 greedy로 하려다가 풀이를 검색해 보고 그리디의 그자도 없는 것을 보고 좌절..

<hr/>

**개념 정리**

대부분 dfs로 풀었습니다. 
dfs로 한다면 backtrack하면서 굉장히 쉽게 풀었을 것 같은데... 왜 괜히 그리디로 한다고..
최근에 알고리즘 수업시간에 greedy구현해서 그런걸까요
